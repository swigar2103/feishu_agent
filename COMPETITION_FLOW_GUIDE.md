# 比赛赛道：多 Agent 成品流程 · 分析与协作指导

本文档基于当前比赛「成品流程图」与队内分工整理，用于 **A（编排）/ B（资源与检索）/ C（生成与展示）** 对齐架构、边界与迭代顺序。**实现代码中资源池目录**为 `src/resource_pool/`（与仓库源码根目录一致；流程说明中的「根目录 resource_pool」指该模块路径）。

---

## 1. 流程图高层结构

整体为 **自上而下** 的流水线，并包含若干 **质量控制与补足** 的反馈环。

| 阶段 | 典型节点（名称以流程图为准） | 主要职责 |
| --- | --- | --- |
| 准入与任务澄清 | 任务入口判断、请求守卫 | 将 IM/飞书中的自然语言变为 **可执行任务**；安全与范围过滤 |
| 候选资料与意图 | 候选资料筛选、任务意图识别、模板匹配、任务规划 | **先粗筛资源** → 识别意图与缺口 → **产出执行计划与目标结构** |
| 深度资料与生成 | 重点资料获取、分析整理、草稿生成 | 从摘要进入 **正文/详情**，再 **分析 → 初稿** |
| 审查与输出 | 风格检查、合规检查、成果输出、展示与同步 | **风格环**（不过则回草稿）、**合规环**（数据/口径问题回分析；结构/核心问题可回规划） |
| 记忆与资源闭环 | 记忆学习、资源池增量 | 更新 **用户偏好/项目记忆**，并把 **高价值线索写回资源池** |

### 1.1 三条关键反馈环（与实现强相关）

1. **资源不足**：候选集不够 → **LLM / 语义 fallback** 补检索（B2），仍不足则由 **Planner 调整路径**（A）。
2. **信息不足**：重点资料仍缺 → **Planner 调整子任务或引导补检索**（A 主流程调度，B 执行检索）。
3. **审稿不通过**：风格不过 → **Writer** 重修；合规不过 → **Analyst / Writer** 或回 **Planner**（依问题类型分叉）。

### 1.2 与 LangGraph 的关系

- **主流程、条件边、callback 汇合** 由 **A** 实现的 **LangGraph** 统一管理。
- **B** 的节点在服务层表现为「供调用的上下文能力」，**不包含** 「下一步去哪个 Agent」的控制逻辑。
- **C** 暴露分析/初稿/最终产物生成，**不负责**编排层的 callback 路由。

## 2. 三角色分工（与成品流程对齐）

### 2.1 A：主流程与 Agent 编排

- Request Guard  
- Intent Agent  
- Skill Router  
- Planner  
- Style Reviewer  
- Compliance Reviewer  
- **LangGraph 主流程**  
- **API 总入口**与最终联调  

### 2.2 B：资源与检索层

| 编号 | 模块 | 对应流程图位置（概念） |
| --- | --- | --- |
| B1 | Resource Pool Manager | 资源池底座；为「候选资料筛选」提供统一池 |
| B2 | Resource Screening | 候选资料筛选；输出 Resource List |
| B3 | Retriever | 重点资料获取；摘要 → 正文/详情 → 任务上下文包 |
| B4 | Resource Pool Enricher | 记忆学习之后的 **资源池增量**；权重与关联 |
| B5 | Feishu / 资源适配（Mock → 真实 API） | 文档摘要、目录、联系人、项目、消息线程等 **统一适配** |

### 2.3 C：内容生成与展示层

- Analyst  
- Writer  
- Output Generator  
- Doc / PPT / Canvas 输出  
- 桌面端 / 移动端展示壳  
- 状态同步展示

## 3. 协作原则（队内约定）

1. **核心接口由 A 先定义**；**B / C 不擅自改字段名**（仅在 A 发起的契约变更通知后对齐版本）。
2. **B**：只对外提供 **候选资源列表** 与 **详细上下文资料包**，**不介入**主流程控制（不写「下一跳是哪个节点」）。
3. **C**：只对外提供 **分析结果、初稿、终稿**，**不控制**编排层的 callback。
4. **callback 路径仅由 A 的 LangGraph** 声明与修改。
5. **所有人仅通过约定接口协作**，不改他组模块内部实现。

---

## 4. B 方向工作包与验收关注点（概要）

### B1 — Resource Pool Manager

- **实现位置**：`src/resource_pool/`（`ResourcePoolManager`、`types.ts`、`mock/*.json`，由 `src/resource_pool/index.ts` 统一导出）。
- **做什么**：四类摘要（文档 / 联系人 / 项目资料 / 用户画像）的统一结构、加载与查询，输出 **可供筛选的 Resource Pool**。
- **输入**：Mock 飞书或外部资源数据（JSON 等）。
- **验收**：四类资源齐全、可查询；结构稳定供 B2/B4 复用。

### B2 — Resource Screening

- **做什么**：规则 / 关键词 / 标签 / 相似度粗筛；不足时用 **LLM fallback** 语义补筛。
- **输出**：**Resource List**（候选集）。

### B3 — Retriever

- **做什么**：按任务计划与候选列表 **深度读取**，形成「任务上下文包」（文档正文、联系人职责、项目详情等；当前可 Mock，但结构贴近真实）。
- **输入**：依赖 A 侧的 **Planner 输出结构**（**待 A 定稿 JSON Schema**）。

### B4 — Resource Pool Enricher

- **做什么**：本次实际使用的高价值资源、用户修改后的信息、更新后的记忆 → **写回池子**，更新权重与资源间关系。

### B5 — 适配层

- **做什么**：`BaseAdapter` 风格接口；现阶段 **MockFeishuAdapter**；未来换真实飞书时不推翻上层检索设计。

---

## 5. 数据流占位（与其它角色对接）

以下为逻辑关系，**具体字段名以 A 发布的契约为准**：

- **User 请求** → A（Guard / Intent / Router / Planner）  
- **Planner 计划 + 用户任务描述** → B2/B3：**Resource List → 上下文包**  
- **上下文包 + 计划** → C（Analyst → Writer → …）  
- **终审结果 + 使用过的资源清单 + 用户反馈 + 记忆更新** → B4：**资源池 & 权重**  

---

## 6. 与当前仓库实现的衔接说明

- 现有图谱与类型见 `src/graph/`、`src/types/contracts.ts`、`src/schemas/index.ts`；往比赛架构迁移时，应 **增量引入** `src/resource_pool/` 与 B2–B5，避免一次性破坏现有 demo。
- **Planner / Retriever 的 JSON 契约**：B3 在 **A 给出正式 Schema** 后再锁版本实现，避免返工。
